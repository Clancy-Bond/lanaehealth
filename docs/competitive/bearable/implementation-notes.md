## Feature 1: Plain-English insight cards with r-value and lag

Convert existing `correlation_results` rows into plain-sentence insight cards on the Patterns page, with optional elevation of the top finding to the Home page. Include correlation coefficient, sample size, lag days, and confidence tier in the card.

### File targets

**New:**
- `src/components/patterns/InsightCard.tsx` (new component)
- `src/lib/api/insights.ts` (new API module: `getTopInsights`, `getInsightById`, `formatInsightSentence`)
- `src/lib/intelligence/insight-narrator.ts` (new: Claude-templated sentence generator; routes through `src/lib/context/assembler.ts`)

**Modify:**
- `src/components/patterns/PatternsClient.tsx` (render new InsightCard list above CorrelationCards)
- `src/components/patterns/CorrelationCards.tsx` (add lag badge if `lag_days` is non-null; keep existing card intact for power users)
- `src/app/patterns/page.tsx` (pass insights fetched on server to PatternsClient)
- Optional: `src/app/page.tsx` (home page hero insight surface)

**No changes to:**
- `src/lib/context/assembler.ts` (use as-is via new insight-narrator)
- Existing Supabase tables

### Data model

**No new tables.** Reads `correlation_results` which already exists (8 rows for Lanae, columns: `factor_a`, `factor_b`, `correlation_type`, `coefficient`, `p_value`, `effect_size`, `effect_description`, `confidence_level`, `sample_size`, `lag_days`, `cycle_phase`, `passed_fdr`, `computed_at`).

**Read-only:** yes. This feature is purely presentational over existing data.

**Narrative embedding:** The generated insight sentences should be chunked per-week and written to `health_embeddings` with `content_type='insight_narrative'` so Chat can retrieve them. Per design rule 6.6.

### Component plan

- `InsightCard`: props `{ factor_a, factor_b, coefficient, sample_size, lag_days, confidence_level, sentence }`. Renders a card with the sentence as the hero line, an r = X.XX chip, a lag badge ("same day" / "next day" / "2 day lag" / "3+ day lag"), a sample size footnote ("based on 42 days"), and a severity/confidence color bar. Use CSS vars: `--accent-sage` for strong, `--accent-blush` for moderate, `--text-secondary` for suggestive.
- Reuse: `EmptyState` from `src/components/ui/EmptyState.tsx` for zero-insights state.
- Reuse: Existing CorrelationCards `CONFIDENCE_STYLES` palette for consistency.
- Typography: `--text-lg` for the hero sentence, `--text-sm` for r-value and sample size.
- Shape: `--radius-lg` corners, `--shadow-sm`.

### Sentence generator

`insight-narrator.ts` accepts a `CorrelationResult` row and returns a sentence via Claude Sonnet 4.6, routed through assembler.ts. Templates to seed:

- Positive strong with lag: "When [factor_a] is higher, [factor_b] tends to be higher about [lag_days] day(s) later, based on [sample_size] days of data."
- Inverse strong: "Days with more [factor_a] tend to have less [factor_b], based on [sample_size] days."
- Suggestive: "There's a possible link between [factor_a] and [factor_b] in your data, though it needs more observation."

Always append confidence tier in small text and never state causation. Static/dynamic boundary: confidence rules and tone in the cached system prompt, factor names and numbers dynamic.

### Acceptance criteria

1. `/patterns` page shows up to 5 InsightCard components above the CorrelationCards section when `correlation_results` has rows.
2. Each InsightCard shows the plain-English sentence, r-value, lag badge, and sample size.
3. The sentence is generated by Claude and cached (no regeneration on every page render).
4. Lag badge displays "same day" for lag_days=0, "next day" for 1, "2 day lag" for 2, "3+ day lag" for 3 or more.
5. Confidence color bar uses the correct CSS var.
6. Zero-state: if `correlation_results` is empty, show a friendly message via `EmptyState` not an error.
7. Home page hero InsightCard (if shipped) pulls the single top-ranked insight.
8. No streak or guilt copy anywhere in the card text.
9. `npm run build` passes.
10. `npm test` passes with at least 3 new tests (sentence template, lag badge logic, empty state).

### Verification plan against Lanae's real data

1. Query `correlation_results` in Lanae's Supabase. Confirm 8 rows exist with `passed_fdr = true` for the 4 strong patterns.
2. Spot-check one known correlation (e.g., the standing pulse to energy pattern if present) and verify the generated sentence reads naturally and matches her reality.
3. Screenshot `/patterns` on port 3005 showing the insight cards.
4. Manual: Lanae's Patterns page should have at least 4 InsightCards visible after implementation.
5. Run `src/lib/intelligence/auto-trigger.ts` (if it exists for correlations) or equivalent `analysis_runs` refresh. Confirm `computed_at` is within 7 days.
6. Verify embedding sync: each generated sentence writes one chunk to `health_embeddings` with `content_type='insight_narrative'`.

### Risks

- Stale correlation_results. If `computed_at` is > 30 days old, sentences may describe patterns no longer present. Mitigation: show "based on data through YYYY-MM-DD" in small footnote.
- Claude overclaiming causation. Mitigation: strict template, unit tests on sentence output shape, always end with confidence qualifier.
- Cycle-phase-specific correlations mislabeled. Mitigation: if `cycle_phase` column is non-null, include "during luteal phase" etc. in the sentence.
- Token cost. Mitigation: cache generated sentences keyed on `(correlation_id, computed_at)`. Regenerate only when `correlation_results` row updates.

## Feature 3: PRN post-dose efficacy polling

When Lanae logs a PRN (as-needed) medication, schedule a push notification 90 minutes later asking if it helped. Capture 2-tap response (Yes helped / No didn't / Too soon to tell). Store as time series for effectiveness trending.

### File targets

**New:**
- `src/lib/migrations/013-prn-efficacy.sql` (new migration; see Data model below)
- `src/lib/api/prn-efficacy.ts` (new: `logDose`, `recordEfficacyResponse`, `getEfficacyTimeSeries`)
- `src/components/log/PrnEfficacyPrompt.tsx` (new: in-app prompt surfaced if user opens app before push fires)
- `src/lib/intelligence/prn-escalation.ts` (new: cross-ref with frequency escalation)
- `src/app/api/push/schedule-efficacy/route.ts` (new: server endpoint that Web Push calls)

**Modify:**
- `src/components/log/MedicationEntry.tsx` or `src/components/log/MedicationCard.tsx` (when a PRN is logged, call `logDose()` which schedules the follow-up)
- `src/lib/api/medication-adherence.ts` (extend to record `is_prn` flag if not already captured)
- `src/components/patterns/MedTimeline.tsx` (overlay efficacy ratings as small dots on the timeline)

**No changes to:**
- `medication_reminders` table (existing)
- `push_subscriptions` table (from migration 012, existing)

### Data model

New table in migration `013-prn-efficacy.sql`:

```sql
-- Migration 013: PRN efficacy polling
-- Additive only. Captures delayed effectiveness ratings for as-needed medications.

CREATE TABLE IF NOT EXISTS prn_dose_events (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  log_id uuid REFERENCES daily_logs(id) ON DELETE SET NULL,
  medication_name text NOT NULL,
  dose_amount text,
  dosed_at timestamptz NOT NULL DEFAULT now(),
  follow_up_scheduled_at timestamptz,
  follow_up_delivered_at timestamptz,
  efficacy_response text CHECK (efficacy_response IN ('helped', 'no_effect', 'worse', 'too_soon', 'skipped')),
  efficacy_responded_at timestamptz,
  notes text,
  created_at timestamptz DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_prn_dose_events_medication ON prn_dose_events(medication_name);
CREATE INDEX IF NOT EXISTS idx_prn_dose_events_dosed_at ON prn_dose_events(dosed_at DESC);
```

**Read-only existing tables:** yes, we only add a new table. `daily_logs` is read-only per design rule.

**Reversibility:** write operations go through `src/lib/api/prn-efficacy.ts`. Any automated transformations log to `analysis_runs` per design rule 6.5.

### Component plan

- `PrnEfficacyPrompt`: modal/bottom-sheet. 3 buttons: "It helped" / "No effect" / "Still too early." Stores response via API.
- Reuse: `BottomNav`, `SaveIndicator` styles for consistency.
- Reuse: `MedicationEntry` for the dose-logging touchpoint.
- Reuse: CSS vars `--accent-sage` for helped, `--accent-blush` for no effect, `--text-secondary` for too early.

On medication log:
1. User logs a PRN dose in MedicationEntry.
2. Frontend calls `logDose()` which inserts a `prn_dose_events` row and POSTs to `/api/push/schedule-efficacy` with a 90-minute delay.
3. If user opens app before push fires and the follow-up is overdue, `PrnEfficacyPrompt` surfaces automatically on next navigation to /log.
4. User taps response; API updates the row with `efficacy_response` and `efficacy_responded_at`.

### Acceptance criteria

1. Migration 013 runs cleanly via `scripts/run-migration.mjs`.
2. When a PRN medication is logged on /log, a `prn_dose_events` row is inserted with `dosed_at` = now() and `follow_up_scheduled_at` = now() + 90 min.
3. Push notification arrives 90 min later (or surfaces in-app if user returns to the app before push fires).
4. User can tap one of 4 responses; `efficacy_response` column is populated.
5. `/patterns` MedTimeline shows efficacy dots overlaid on dose events.
6. Time series of "average efficacy of [med] over last 30 days" is computable via `getEfficacyTimeSeries()`.
7. No new user-facing streak or guilt copy.
8. `npm run build` and `npm test` pass.

### Verification plan against Lanae's real data

1. Check Lanae's current PRN medications via `medication-adherence.ts`. Identify at least one PRN (likely ibuprofen or a breakthrough med).
2. Log a test dose through the /log UI on port 3005. Confirm `prn_dose_events` row appears in Supabase.
3. Wait 90 minutes or use a testing override to trigger the follow-up immediately. Confirm push delivery to Lanae's device (via existing push_subscriptions entries).
4. Tap "It helped." Confirm row updated and `analysis_runs` logs the response event.
5. After 2 weeks of real use, query the time series and confirm at least 5 data points accumulate.
6. Cross-ref with pain_score in daily_logs: verify that doses flagged "helped" correspond to pain_score drops later that day.

### Risks

- Push delivery unreliable on iOS PWA. Mitigation: in-app prompt fallback when user returns to app post-dose-window.
- User annoyance at follow-up prompts. Mitigation: allow per-medication opt-out via settings.
- 90-minute window wrong for some meds (e.g., Tylenol peaks at 60 min, sumatriptan at 30 min). Mitigation: allow per-medication override of the follow-up delay. Default 90.
- Double-logging if user logs the same dose twice. Mitigation: unique index on `(medication_name, dosed_at)` with small time bucket, or toast "already logged."
- Interaction with existing medication_adherence writes. Mitigation: `prn-efficacy.ts` is additive, does not mutate adherence. Adherence remains source of truth for "did they take it"; efficacy is orthogonal.
